Define os papéis especializados de IA usados pelo framework do pedido-backend. Cada arquivo de persona dá ao modelo uma identidade distinta, CoT protocol obrigatório, playbook passo-a-passo, critérios de handoff e red lines — de modo que diferentes padrões de raciocínio sejam ativados para diferentes tipos de trabalho.

## Summary

- maestro.md — persona orquestradora; gerencia boot de sessão, decomposição de tarefas, roteamento de modelo e dispatch
- architect.md — persona de planejamento; produz planos de implementação a partir de requisitos sem escrever código
- coder.md — persona de implementação; escreve e modifica código seguindo o plano do architect
- reviewer.md — persona de code review; avalia mudanças para correctness, conformidade com padrões e risco
- tester.md — persona de testes; escreve testes E2E com Vitest + Testcontainers para a API; verifica cobertura de happy path, error paths e limites de regras de negócio
- security.md — persona de auditoria de segurança; audita arquivos alterados para vazamentos de PII em logs, falhas de auth JWT, multi-tenancy e segredos

## Constraints

- Cada persona DEVE ter estas seções: Identity, CoT Protocol, Playbook, Handoff, Red Lines, Yield
- Cada arquivo de persona DEVE incluir frontmatter com campos obrigatórios: `shortDescription`, `modelTier`, `version`, `lastUpdated`; opcional: `preferredModel`
- Nomes de arquivo DEVEM ser lowercase (ex: `reviewer.md`)
- Fluxo CoT DEVE seguir: `[THINKING] → Playbook Execution → [VERIFICATION] → Handoff`

## Guidance

- SHOULD manter seções de Identity com personalidade vívida, não descrição de cargo
- SHOULD referenciar skills e rules com `(uses: path)` ou `(follows: path)` nos passos de Playbook
