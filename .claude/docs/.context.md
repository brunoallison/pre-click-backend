Referência completa do pedido-backend para consumo por personas. Cada arquivo cobre um domínio distinto; personas devem ler apenas o que é relevante para a tarefa em vez de carregar tudo no contexto.

A verdade canônica de domínio (BASE Adidas, formato Click, regras de negócio) vive no pai em `../../.claude/`. Estes docs cobrem a camada de implementação: como o backend materializa essas regras.

## Summary

- architecture.md — estrutura de módulos, padrão Routes→Tasks, DI com tsyringe, TypeORM DataSource, multi-tenancy via JWT, schema Postgres multi-tenant
- business-rules.md — regras de negócio implementadas: Qty zero não exporta, RDD imutável, VOL MÍNIMO por DIVISION, cluster de loja, CUSTOMER imutável após atribuído
- coding-standards.md — ESLint, Prettier, Conventional Commits, proibição de `process.env` direto, mensagens de validação em pt-BR
- project-context.md — visão geral da API, scripts npm, módulos, convenções de rota e autenticação
- testing.md — stack de testes (Vitest, supertest, Testcontainers), estrutura de arquivos, padrão de isolamento por teste, auth via JWT nos testes
- api-contract.md — endpoints, DTOs, respostas e headers; fonte de verdade local para o contrato HTTP (o contrato canônico vive em `../../.claude/docs/backend.md`)

## Constraints

- MUST NOT editar migrations existentes — criar nova migration para qualquer correção de schema
- MUST NOT usar `process.env` diretamente — sempre `src/config/env.ts` ou `src/config/migration-env.ts`
- MUST atualizar `src/docs/openapi.ts` após qualquer mudança em endpoint, DTO ou resposta
- MUST NOT derivar `tenantId` do request body — sempre extrair do JWT via middleware

## Guidance

- Architect e Reviewer: priorizar `business-rules.md` e `architecture.md`
- Coder: `architecture.md` + `coding-standards.md` antes de implementar
- Tester: `testing.md`; `business-rules.md` para validar cenários de borda
- Security: `architecture.md` (fluxo de auth) + `api-contract.md` (superfície de ataque)
